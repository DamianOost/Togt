## Where Togt actually is right now

A working two-sided marketplace POC with three things shipped that no other SA gig app has:

1. **Auto-match flow** — tap once, dispatcher pings nearby labourers in priority order, first to accept produces a booking. Race-safe under concurrent accepts. Live on phone.

2. **Agent-native API surface** — RFC 9457 errors, Idempotency-Key middleware, OpenAPI 3.1 at `/.well-known/openapi.json`, scoped Stripe-style API keys, MCP server with both stdio (Claude Desktop) and HTTP/StreamableHTTPServerTransport at `/mcp`.

3. **VerifyNow real DHA KYC** — sandbox mode for now. Production toggle is one .env flip.

What this means: an AI agent (Claude, Operator, A2A orchestrator) can connect to Togt and book a labourer end-to-end without any custom integration code. That capability does not exist anywhere else in SA's gig marketplace landscape today.

## What's working

- Auth: register / login / refresh-rotation / logout / forgot-password / reset-password
- 401 auto-refresh interceptor on mobile (silent token rotation, no user-visible interruption)
- KYC: VerifyNow `said_verification` with structural-Luhn fallback
- Bookings: full state machine with 3-push notification chain (matched → in_progress → completed)
- Payments: Peach customer-side checkout + HMAC webhook verification
- Auto-match: customer screen + labourer modal, 7 backend endpoints, Socket.io `/match` namespace, in-memory dispatcher
- API surface: 14 paths in OpenAPI, 11 stable error types, idempotency on 4 mutating endpoints
- MCP: 10 tools (find_labourers, estimate_booking_cost, create/get/cancel_match_request, list_my_bookings, get_booking, admin_stats, force_expire_match, marketplace_stats)
- Decision-context fields on every MCP tool result: review_count, acceptance_rate_pct, completion_rate_pct, days_since_last_booking, cancellation_window_seconds, expected_completion_rate_pct
- Backend at 84/84 tests, ~13s suite, three consecutive cold runs green (no flake)

## What ships next, ranked

### P0 — strategic continuation of Path B

These complete the agent-native bet. All three were demanded across the introspection memos:

1. **Webhooks for booking lifecycle events.** Memo 3's "single highest-leverage addition." Stripe pattern: registration endpoint, HMAC signatures via `X-Togt-Signature`, retry with exponential backoff (30s, 2m, 10m, 1h, dead-letter at 24h), event payload includes full resource state. ~1 day. Schema: new `webhook_subscriptions` table + `webhook_deliveries` table.

2. **Audit log primitive.** Every MCP tool call gets logged with `agent_id` (from API key), `action`, `resource`, `result`, `timestamp`. New `audit_log_query(agent, action, since)` admin tool. Memo 2's "marketplace won't trust me if it can't audit me." ~half day.

3. **Get_booking_full_context** — one call returns booking + labourer reliability + customer history + messages + GPS pings. Memo 2's #1 want for ops. ~half day.

### P2 — pure code, no vendor (can ship without Damian)

7. **Offline persistence on mobile.** Cache active bookings/matches in AsyncStorage so they survive load-shedding-induced network drops. Show "OFFLINE" badge with last-known state. ~1 evening.

8. **Pin-it location UX.** Customer drops a pin on the map at booking time, worker navigates to the pin not the address — eliminates "I couldn't find your house" cancellations endemic to informal-settlement addresses. ~1 evening.

9. **isiZulu locale framework.** Set up i18next or react-i18next, extract strings from worker-facing screens, ship empty translations file. Damian then commissions a native speaker (R3-5k once-off) to fill it in. Setup is ~half day; the human translation review is the real cost.

10. **Migrate remaining route handlers to ProblemError.** auth.js, bookings.js, kyc.js etc still emit `res.status(N).json({error: X})` style. The mobile app reads `err.response.data.error` (still works thanks to backward-compat field) but agents would benefit from full RFC 9457. ~half day, mechanical refactor.
